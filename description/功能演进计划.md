## Phase 8：RSS 图片抓取（已完成）

### 实现方案

**Go 层：** `backend-go/utils/rss_parser.go` 新增 `ExtractImageURLs()` 函数，在 `CleanContent()` 清洗 HTML 之前提取所有 `<img src>` URL，存入 `FeedItem.ImageURLs []string`。

**数据库：** `content` 表新增 `image_urls JSONB DEFAULT '[]'` 字段。

**前端：** `Reader.vue` 渲染 `image_urls` 数组图片，懒加载 + 失败占位。

### 实际效果

```go
// rss_parser.go
func ExtractImageURLs(html string) []string { ... }
// 在 SanitizeFeedItem() 里调用：
item.ImageURLs = ExtractImageURLs(rawHTML)  // 清洗前提取
item.Content = CleanContent(rawHTML)         // 再清洗纯文本
```

## Phase 9：RSS 作者过滤（已完成）

### 实现方案

`sources` 表新增 `author_filter JSONB DEFAULT '{}'`，支持白名单/黑名单：

```json
{"whitelist": ["作者A"], "blacklist": ["作者B", "作者C"]}
```

Go 的 `rss_service.go` 在 `processItem()` 入队前检查作者：

```go
if !passesAuthorFilter(article, source.AuthorFilter) {
    return  // 不匹配则跳过，不入 Redis Stream
}
```

过滤在去重之前，节省 Redis 和 LLM 的开销。

## Phase 10：智能偏好学习（已完成）

### 完整链路

```
用户自然语言对话（Bot 或前端）
    ↓
ReAct Agent 调用 update_preferences 工具
    ↓
tools._update_preferences(db_pool) 增量合并写入 user_preferences 表
    ↓
stream_consumer 每次评估前调用 _inject_preferences()
    ↓
读取全局偏好 / 源级别偏好（源级别优先）
    ↓
format_preferences_for_prompt() 格式化为自然语言段落
    ↓
动态拼接到评估 system_prompt 末尾 → LLM 打分受偏好影响
```

### 偏好数据结构（JSONB，增量合并）

```json
{
  "liked_topics": ["AI", "系统设计", "时事"],
  "disliked_topics": ["营销", "Rust"],
  "min_innovation_score": 6,
  "custom_notes": "偏好有代码示例的技术深度文章"
}
```

### 关键实现细节

- **增量合并**：数组字段（liked_topics / disliked_topics）追加去重，不覆盖；标量字段覆盖
- **两级偏好**：先查 `source_id` 对应的源级偏好，无则回退全局偏好（`source_id IS NULL`）
- **实时生效**：偏好写入后，下次评估即生效，无需重启任何进程
- **已解决 Bug**：`preference_tools._db_pool` 全局变量仅在 `api_server` 启动时设置，Telegram Bot 进程中为 None 导致静默失败。修复：`_update_preferences` 工具直接使用传入的 `db_pool` 参数，完全绕过全局变量

## Phase 11：后台通知系统（已完成）

### 实现的三个子系统

**11.1 站内通知（SSE）**

- `notifications` 表：存储高分文章通知记录
- Python `stream_consumer` 评估完成后写入 notifications 表，同时通过 Redis Pub/Sub 发布事件
- Go 后端 `GET /api/notifications/stream` 订阅 Redis，通过 SSE 推送给前端
- 前端 `NotificationBell.vue`：导航栏铃铛 + 未读数角标 + 下拉列表

**11.2 Telegram 推送（主动推送）**

- `notification_settings` 表：配置推送渠道（JSONB 数组，支持多渠道扩展）
- `push_service.py`：评估完成后调用 Telegram Bot API 发送 MarkdownV2 格式消息
- 推送条件：`notify_on_interesting=true` 且 decision=INTERESTING，或评分超过阈值

**11.3 Telegram Bot 双向控制**

Bot 作为独立第三个 Python 进程运行：

| 功能 | 实现 |
|------|------|
| `/status` | HTTP → Go API 获取各状态文章数 |
| `/fetch` | 并发触发所有 RSS 源立即抓取 |
| `/recent` | asyncpg 查询最近 5 篇 INTERESTING 文章 |
| 自然语言 | 接入 ReAct Agent（与前端共用同一套）|
| 白名单鉴权 | chat_id 集合校验，未授权静默丢弃 |
| 限流 | 每用户每秒最多 1 条 |
| 任务队列 | 每用户独立 asyncio.Queue，串行消费防并发 |
| 上下文记忆 | 内存 histories dict，保留最近 10 轮（20条）|

### 关键技术难点（已解决）

**Event loop 错配导致 /fetch、/recent 持续报错：**

`asyncio.run(_setup())` 创建并关闭 event loop A，asyncpg pool 绑定于 A。`app.run_polling()` 创建 event loop B，在 B 里使用绑定于 A 的 pool 触发 `cannot perform operation: another operation is in progress`。

解决：使用 `Application.builder().post_init(callback)` 在 PTB 自己的 event loop 内创建 pool；启动配置通过一次性临时连接读取，用完立即关闭。


### 服务器部署方案（备忘）

各服务用 `systemd` 管理，配置模板：

```ini
# /etc/systemd/system/junkfilter-bot.service
[Unit]
Description=Junk Filter Telegram Bot
After=network.target postgresql.service

[Service]
WorkingDirectory=/home/deploy/Junk-Filter/backend-python
ExecStart=/path/to/venv/bin/python telegram_bot.py
Restart=always       # 崩了自动重启
RestartSec=3
EnvironmentFile=/home/deploy/Junk-Filter/.env

[Install]
WantedBy=multi-user.target
```

```bash
systemctl enable junkfilter-bot   # 开机自启
systemctl start junkfilter-bot    # 立即启动
systemctl status junkfilter-bot   # 查看状态
journalctl -u junkfilter-bot -f   # 实时查看日志
```

每个服务（Go、Python API、Consumer、Bot）各建一个 `.service` 文件，Docker 负责 PostgreSQL + Redis，其余服务用 systemd 管理。部署后 Telegram Bot 的 macOS App Nap 问题自然消失。
