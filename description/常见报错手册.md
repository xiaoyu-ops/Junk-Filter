## 1. Format mismatch: only openai_responses allowed

**完整错误**
```
Error code: 500 - {'error': {'message': 'Format mismatch: Request appears to be in
format [openai_chat, claude_chat], but only [[openai_responses]] is allowed.',
'type': 'moderation_error', ...}}
```

**根本原因**

你选择的模型（如 o3、o4-mini 等新模型）只支持 OpenAI **Responses API**（`/v1/responses`），不支持旧的 Chat Completions API（`/v1/chat/completions`）。系统默认走 Chat Completions，中转站检测到格式不匹配报错。

**发生场景**

- 前端 Agent 对话页发消息
- 后台 stream_consumer 自动评估文章

**解决方法**

无需手动操作。系统会在第一次失败时自动检测并切换到 Responses API，后续请求直接走新路径，不再报错。如果持续报错，检查中转站是否支持该模型的 Responses API。

## 2. LLM审核并发数超限 (max=5)

**完整错误**
```
Error code: 500 - {'error': {'message': 'LLM审核并发数超限(max=5)',
'type': 'moderation_error', 'code': 'MODERATION_BLOCKED'}}
```

**根本原因**

短时间内发送了多条消息，触发了多个并发 LLM API 调用，超过中转站允许的最大并发数。

**发生场景**

- 前端 Agent 对话页快速连续点击发送（每次报错后 `isLoading` 重置，可以继续发送）
- 后台评估任务 + 前端聊天同时打到同一中转站

**解决方法**

系统已在后端加入串行队列（`asyncio.Semaphore(1)`），同时只有一个 LLM 调用。后来的请求会等待，而不是并发发出。如果仍然触发，检查 `api_workers` 是否为 1（多 worker 会绕过 semaphore）。

```bash
# 确认只有 1 个 worker 进程
lsof -i :8083 | grep LISTEN  # 应该只有 1 行
```

## 3. LLM 服务暂时不可用（504 网关超时）

**完整错误**（前端或 Telegram Bot）
```
LLM 服务暂时不可用（504 网关超时），请稍后重试
```

**根本原因**

LLM API 请求超过中转站或上游服务的超时时间（通常 30～60 秒无响应），中转站返回 504。常见于：
- 中转站负载高
- 所选模型响应慢（推理模型如 o3 首次响应较慢）
- 网络波动

**发生场景**

- 前端 Agent 对话
- Telegram Bot 自然语言问答

**解决方法**

稍等片刻重试。如果频繁发生：
- 换响应更快的模型
- 在配置中心检查 `base_url` 是否指向稳定的中转站
- 联系中转站确认服务状态

## 4. 服务繁忙，请稍后重试（前一个请求仍在进行中）

**完整错误**（前端 Agent 对话）
```
服务繁忙，请稍后重试（前一个请求仍在进行中）
```

**根本原因**

系统串行队列中已有一个 LLM 调用在进行，新请求等待超过 30 秒仍未获得执行权。通常因为上一个请求的 LLM 调用挂住（如网络中断后连接僵死）。

**发生场景**

- 前一个请求遇到 504 超时但连接未正常关闭，锁被长时间占用
- 期间又发了新消息

**解决方法**

等待约 30 秒后重试。如果反复出现，重启 Python API 服务器清除僵死连接：

```bash
kill $(lsof -t -i :8083)
conda activate junkfilter && cd backend-python && python api_server.py
```

## 5. Bot 不回应消息

**现象**

发消息给 Telegram Bot，没有任何反应（连"思考中…"都没有）。

**根本原因**

你的 Telegram chat_id 不在白名单（`allowed_ids`）里，`_auth` 装饰器静默丢弃了消息。

**排查方法**

查看 bot 进程日志，看是否有：
```
[Bot] Unauthorized: <你的chat_id>
```

**解决方法**

在前端配置中心「消息推送」页面确认填写的 `chat_id` 与 Telegram 实际 chat_id 一致。Bot 启动时会自动把 DB 里的 `chat_id` 加入白名单。如需添加其他账号，在 `telegram_bot.py` 的 `allowed_ids` 里手动加入对应 ID，重启 Bot。

## 6. Bot 空闲一段时间后假死

**现象**

长时间（1 小时以上）不和 Bot 交互后，再发消息没有回应，或命令无反应。

**根本原因**

Bot 与数据库之间的连接池在长时间空闲后，PostgreSQL 服务端会主动关闭这些连接。Bot 下次使用时拿到的是已断开的"死连接"，操作报错，消息处理卡住。

**解决方法**

已修复：连接池配置了 `max_inactive_connection_lifetime=300`，任何超过 5 分钟未使用的连接会自动重建。同时添加了 PTB 网络超时和 Worker 兜底超时（150 秒），确保单条消息卡住不会堵死整个队列。

如果重启后仍出现，检查 PostgreSQL 的 `idle_in_transaction_session_timeout` 配置是否比 300 秒短。

## 快速对照表

| 错误关键词 | 原因 | 操作 |
|-----------|------|------|
| `openai_responses` | 模型只支持 Responses API | 自动修复，重试即可 |
| `并发数超限` | 多个并发请求 | 等 1 秒再发，检查 worker 数量 |
| `504` | 中转站/模型响应超时 | 稍后重试或换模型 |
| `服务繁忙` | semaphore 等待超时 | 等 30 秒或重启 api_server |
| Bot 无响应 | chat_id 不在白名单 | 确认 chat_id 配置正确 |
| Bot 假死 | DB 连接池 stale 连接 | 已自动修复；仍出现则重启 Bot |
