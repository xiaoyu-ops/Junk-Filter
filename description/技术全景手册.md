# Junk Filter — 技术全景手册

> 目标：快速掌握整个项目的技术选型、设计决策和实现深度。

---

## 一、项目定位

个人 RSS 内容过滤系统。核心问题：订阅源每天产出数百条资讯，90% 是噪音，人工筛选效率极低且偏好难以规则化表达。

解决方案：构建一条从抓取到评估到交互的全自动管道，LLM 替代人工判断，自然语言替代配置界面。

**规模感知**（本地运行）：
- 73 个 RSS 源，每 30 分钟全量抓取一次
- 单次抓取并发协程数：最多 50 个（协程池限流）
- LLM 评估：每篇文章约 3-10 秒，串行消费（LLM 为瓶颈）
- DB 存储：content 表 + evaluation 表，URL 唯一约束

## 二、整体架构

```
互联网 RSS 源（73个）
      │
      ▼
[Go] 抓取层（并发协程池 + 三级去重 + 质量过滤）
      │ XADD（内容 + 元数据）
      ▼
Redis Stream（ingestion_queue，消费者组）
      │ XREADGROUP（至少一次语义）
      ▼
[Python] LangGraph 评估状态机（有状态图 + 自动重试）
      │
      ├──► PostgreSQL（content + evaluation 持久化）
      │
      ├──► Telegram 推送（push_service → Bot API）
      │
      └──► ReAct Agent（自然语言查询/管理）
                 ↑
          前端对话页（SSE 流式）
          Telegram Bot（长轮询 + 任务队列）
```

**为什么这样分层？**

| 选择 | 原因 | 替代方案及为什么没选 |
|------|------|------|
| Go 做抓取 | goroutine 天然适配 I/O 密集并发；Bloom Filter 在 Go 里有极好的库支持 | Python asyncio 也能做，但 Go 的并发模型更轻量，内存占用更低 |
| Redis Stream 解耦 | 抓取速率（毫秒级）与 LLM 评估速率（秒级）差距 1000 倍，Stream 作弹性缓冲 | 直接 DB 队列：轮询开销大；Kafka：运维复杂，个人项目杀鸡用牛刀 |
| Python 做评估 | LangChain/LangGraph 生态成熟；asyncpg 异步驱动适配 LLM 长等待 | Go 调 LLM：工具链不成熟，ReAct 需要自己实现所有状态机 |
| SSE 不用 WebSocket | 推理输出是单向流，SSE 更简单，无需维护双向状态，天然支持断线重连 | WebSocket：双向通信代价，对单向流是过度工程 |
| PostgreSQL + JSONB | UNIQUE 约束做幂等；JSONB 字段灵活扩展推送渠道类型无需改表 | MySQL：JSONB 支持弱；MongoDB：事务弱，UNIQUE 约束弱 |

## 三、Go 抓取层（深度解析）

### 3.1 三级去重机制

这是整个抓取层最核心的设计，目的是在高并发下以最低代价拒绝重复内容。

```
新 URL 进来
  │
  ├─► L1: 内存 Bloom Filter
  │     启动时从 DB 预热近 7 天的 URL hash（warm-up）
  │     误判率 < 0.1%，零 I/O，纳秒级响应
  │     误判（把新内容当重复）：极少，可接受
  │     不会漏判（把重复当新）：Bloom Filter 数学保证
  │
  ├─► L2: Redis Set（SETNX + TTL 7天）
  │     精确校验，O(1) 查询，~0.1ms
  │     处理 L1 放过的少数漏网（约 0.1%）
  │
  └─► L3: PostgreSQL UNIQUE 约束
        最终防线，捕获高并发下 L1/L2 均通过的极端竞态
        （两个协程同时通过前两层，INSERT 时 DB 保证唯一）
        捕获 INSERT 冲突后 graceful 忽略，不算错误
```

**为什么需要三层？**

单用 DB UNIQUE：每条 URL 都打 DB，高并发下性能差。
单用 Bloom Filter：有误判且无法处理并发竞态（两协程同时查询都返回"不存在"）。
三层组合：99.9% 的重复在 L1 纳秒级拦截，剩余的在 L2 精确拦截，极端竞态由 L3 兜底。

**Bloom Filter 实现细节：**
- 启动时加载近 7 天 content 的 URL，预热 Filter
- 使用多个哈希函数降低碰撞概率
- 不持久化（每次启动重建），简单且正确

### 3.2 并发协程池

```go
// 每个 RSS 源并发抓取，但全局限制协程数
semaphore := make(chan struct{}, 50)  // 最多 50 并发

for _, source := range sources {
    semaphore <- struct{}{}
    go func(s Source) {
        defer func() { <-semaphore }()
        fetchSource(s)
    }(source)
}
```

为什么用 semaphore 而不是 worker pool？semaphore 更简单，任务天然边界清晰（一个源一个任务），不需要预建 worker。

### 3.3 入队前质量过滤

在进入 Redis Stream 之前做两道过滤，节省 LLM Token 消耗：

```go
// 短内容过滤（摘要类预告文章，无实质内容）
if len([]rune(content)) < 200 {
    return // 不入队
}

// 作者过滤（每个源支持 JSONB 白/黑名单）
// 白名单：只接受指定作者
// 黑名单：排除指定作者
if !passesAuthorFilter(article, source.AuthorFilter) {
    return
}
```

`author_filter` 存在 sources 表的 JSONB 字段，格式：
```json
{"whitelist": ["作者A"], "blacklist": ["作者B", "作者C"]}
```

### 3.4 Favicon 自动推导

添加 RSS 源时自动构造图标地址，无需手动维护：

```go
// source_repo.go Create()
```

## 四、Python LLM 评估层（深度解析）

### 4.1 LangGraph 评估状态机

不用简单的"调一次 LLM 拿结果"，而是构建一个有状态的评估图，支持解析失败自动重试：

```
EvaluationState {
    title, content, url    # 输入
    raw_output             # LLM 原始输出（streaming 拼接）
    parsed_result          # 解析后的结构化结果
    retry_count            # 已重试次数
    error                  # 最后一次错误信息
}

节点：
  evaluate_node  →  调用 LLM，强制 streaming=True 收集完整输出
  parse_node     →  从 raw_output 提取 JSON，验证 6 个字段完整性
  should_retry   →  条件边：解析失败且 retry_count < 3 → 回 evaluate_node
                            解析失败且达上限 → 标记 DISCARDED
                            解析成功 → 结束，写 DB
```

**为什么强制 `streaming=True`？**
部分第三方 relay 在非流式模式下只返回 `reasoning` 字段，`content` 为 null（推理模型特有行为）。强制流式后逐 chunk 拼接，可绕过此问题，兼容更多中继服务。

**为什么用 LangGraph 而不是简单 for 循环？**
LangGraph 的状态图让重试逻辑可视化、可测试，状态明确不会出现隐式副作用。实际上 LangGraph 的条件边等价于带状态的 while 循环，但结构更清晰。

### 4.2 评估维度

每篇文章输出 6 个字段，LLM 以 JSON 格式返回：

| 字段 | 类型 | 说明 |
|------|------|------|
| `innovation_score` | 0-10 | 技术新颖性 |
| `depth_score` | 0-10 | 内容深度 |
| `decision` | enum | INTERESTING / BOOKMARK / SKIP |
| `tldr` | text | 一句话摘要 |
| `reasoning` | text | 评估推理过程（供用户理解为何这样打分） |
| `key_concepts` | array | 关键概念标签 |

`decision` 逻辑：
- `INTERESTING`：高分且内容有独特见解，触发 Telegram 推送
- `BOOKMARK`：有价值但不紧急，静默保存
- `SKIP`：噪音，记录但不展示

### 4.3 偏好零样本注入（Prompt Engineering 替代 Fine-tuning）

用户偏好存 `user_preferences` 表，每次评估前动态拼入 system prompt：

```python
# stream_consumer._inject_preferences()
source_id = await get_source_id_for_content(content_id)
prefs = await _get_preferences_from_db(source_id)  # 先查源级别偏好
if not prefs:
    prefs = await _get_preferences_from_db(None)    # 回退到全局偏好
pref_text = format_preferences_for_prompt(prefs)
if pref_text:
    self.evaluator_agent.system_prompt = base_prompt + pref_text
else:
    self.evaluator_agent.system_prompt = base_prompt  # 无偏好时恢复基础 prompt
```

偏好格式（存 JSONB，增量合并）：
```json
{
  "liked_topics": ["AI", "系统设计", "时事"],
  "disliked_topics": ["营销", "Rust"],
  "min_innovation_score": 6,
  "custom_notes": "偏好有代码示例的技术深度文章"
}
```

**偏好更新链路（完整）：**
```
用户对 Bot/前端说「我对时事感兴趣，不感兴趣 Rust」
  ↓
ReAct Agent 调用 update_preferences 工具
  ↓
tools._update_preferences(db_pool=pool) 读取现有偏好
  ↓
增量合并（数组追加去重，标量覆盖）
  ↓
写入 user_preferences 表（WHERE source_id IS NULL 全局偏好）
  ↓
stream_consumer 下次评估时读取 → 注入 system prompt → 影响打分
```

**核心优势：无需重训练**，一次自然语言对话即可改变后续所有评估的偏好权重。

**实现难点：** `update_preferences` 工具需要 `db_pool` 参数，而 `preference_tools.py` 里有个全局 `_db_pool` 变量，仅在 `api_server.py` 启动时设置。Telegram Bot 进程从未调用 `set_db_pool()`，导致 Bot 里偏好更新静默失败。修复：工具实现直接使用传入的 `db_pool` 参数，不依赖全局变量。

### 4.4 Stream Consumer 架构

```python
# 消费者主循环（保证至少一次处理）
while True:
    # block=1000：无消息时阻塞等待最多 1 秒，避免空转 CPU
    msgs = await redis.xreadgroup(GROUP, CONSUMER, {STREAM: ">"}, block=1000)
    
    await _requeue_pending_content()  # 先处理之前崩溃未完成的文章
    await evaluate_batch(msgs)        # 再处理新消息
    await redis.xack(STREAM, GROUP, msg_id)  # 处理完再 ACK
```

**关键设计：先处理后 ACK（至少一次语义）**
Consumer 崩溃时，未 ACK 的消息留在 pending list，重启后重新投递。代价是可能重复处理同一篇文章，通过 DB UNIQUE 约束保证幂等（重复 INSERT 被忽略）。

**`_requeue_pending_content` 的作用：**
每次循环开头检查 DB 里 status=PROCESSING 且 eval_attempts<3 的文章，重新投入评估。这处理的是 Consumer 中途崩溃、Redis ACK 丢失但 DB 状态已更新的边界情况。

### 4.5 eval_attempts 防死循环

```
PENDING → PROCESSING（开始评估）→ EVALUATED（成功）
                    ↘ DISCARDED（eval_attempts >= 3，LLM 持续失败）
```

`eval_attempts` 每次评估失败递增，达到阈值自动 DISCARDED，防止一篇文章无限占用队列资源（比如 LLM 无法解析的格式异常文章）。

### 4.6 LLM 配置热加载

```python
# stream_consumer 每 60 秒检查一次
async def _maybe_reload_llm_config(self):
    new_config = await load_llm_config_from_db(self.db_pool)
    # 只有配置真正变化时才重建客户端，避免无效重建
    if (new_config.get('model_name') != self.current_config.get('model_name') or
        new_config.get('api_key') != self.current_config.get('api_key') or
        new_config.get('base_url') != self.current_config.get('base_url')):
        self._rebuild_llm_client(new_config)
```

## 五、ReAct Agent（深度解析）

### 5.1 自行实现 ReAct，不依赖 Agent SDK

**为什么不用 LangChain Agent 或 Anthropic Agent SDK？**
需要兼容任意 OpenAI Function Calling 协议的第三方接口（GLM、DeepSeek、各类 relay），同时需要自定义 SSE 流式输出格式推给前端。Agent SDK 封装了太多，很难控制中间步骤的输出格式。

```python
# ReAct 核心循环（简化版）
for iteration in range(MAX_ITERATIONS):  # 最多 8 轮，防无限循环
    stream = await client.chat.completions.create(
        model=model,
        messages=messages,         # 累积的完整对话历史
        tools=TOOL_DEFINITIONS,    # OpenAI function calling 格式
        tool_choice="auto",        # 让 LLM 自己决定是否调工具
        stream=True,               # 强制流式
    )

    text_content, tool_calls_map = "", {}
    async for chunk in stream:
        if delta.content:
            text_content += delta.content
            yield SSE("chunk", delta.content)   # 立即推送给前端（打字机效果）
        if delta.tool_calls:
            collect_tool_calls(tool_calls_map, chunk)  # 收集工具调用

    if not tool_calls_map:
        yield SSE("done")
        return                    # 无工具调用 = 最终文字回答，结束

    # 有工具调用：执行 → 追加结果 → 继续循环
    messages.append(assistant_message_with_tool_calls)
    for tc in tool_calls_map.values():
        result = await execute_tool(tc.name, tool_args, db_pool=db_pool)
        yield SSE("tool_call", {tool, args, result})   # 推送工具调用可视化
        messages.append(tool_result_message(tc, result))
```

### 5.2 五个工具的实现细节

| 工具 | 实现 | 关键细节 |
|------|------|------|
| `query_articles` | asyncpg 直连 DB | 参数化查询防 SQL 注入；支持关键词/评分/时间/决策过滤；HAVING 子句做评分聚合过滤 |
| `get_pipeline_status` | HTTP → Go API | 跨服务调用，Go 侧聚合各状态计数，Python 不直接查这张表 |
| `add_source` | HTTP → Go API | 通过 Go API 添加，触发 Go 的 favicon 自动推导逻辑 |
| `remove_source` | HTTP → Go API | Go 处理级联删除（sources → content → evaluation） |
| `update_preferences` | asyncpg 直连 DB | 增量合并（数组字段追加，标量字段覆盖），实时影响后续评估的 system prompt |

**SQL 注入防护（`query_articles`）：**
```python
# 参数化查询，$1/$2 占位符，asyncpg 自动转义
conditions.append(f"(c.title ILIKE ${idx} OR e.tldr ILIKE ${idx})")
params.append(f"%{keyword}%")
rows = await db_pool.fetch(query, *params)  # 参数分离，安全
```

### 5.3 全链路 SSE 流式

```
AsyncOpenAI(stream=True)
  → 每个 text token 立即 yield {"type":"chunk","content":"..."}
  → 工具调用执行后 yield {"type":"tool_call","tool":"...","args":{},"result":{}}
  → 完成 yield {"type":"done"}
  → 出错 yield {"type":"error","error":"..."}

前端消费（useSSE.js）：
  fetch + ReadableStream（不用 EventSource，因为需要 POST 带 body）
  → chunk → 打字机追加到消息气泡
  → tool_call → AgentSteps 面板展开，显示工具调用过程
  → done / error → 停止渲染
```

**为什么用 `fetch` + `ReadableStream` 而不是 `EventSource`？**
`EventSource` 只支持 GET 请求，无法携带 body。我们需要在请求 body 里传 LLM 配置（model、api_key、base_url），所以用 `fetch` 手动消费流。

## 六、Telegram Bot（深度解析）

### 6.1 架构设计

Bot 是第三个独立 Python 进程，与 api_server 和 stream_consumer 共享 DB，但进程完全独立：

```
telegram_bot.py
  ├── python-telegram-bot v21+（异步原生，长轮询）
  ├── asyncpg 连接池（min=5, max=20，在 PTB event loop 内创建）
  ├── 启动时临时连接读取 bot_token + chat_id + llm_config
  └── post_init 回调创建 DB pool，post_shutdown 关闭
```

**为什么用长轮询不用 webhook？**
私人 Bot 无需高并发，长轮询部署更简单，无需公网 IP 和 HTTPS 证书。

### 6.2 三层防护

```
消息进来
  │
  ├─► 白名单校验（_auth 装饰器）
  │     chat_id 不在 allowed_ids → 静默丢弃（不回复，防探测）
  │
  ├─► 限流（_auth 装饰器）
  │     同一用户 1 秒内第二条 → 静默丢弃
  │     _last_message_time dict 记录各用户最后消息时间
  │
  └─► 任务队列（handle_message）
        每用户独立 asyncio.Queue（maxsize=10）
        独立 worker task 串行消费，避免同一用户并发抢 DB 连接
        队列满 → 回复「请稍后再发」
```

### 6.3 Event Loop 错配问题及解决（关键难点）

**问题根源：**

python-telegram-bot v21+ 的 `run_polling()` 内部调用 `asyncio.run()`，会创建一个新的 event loop。如果 asyncpg pool 在外部的 `asyncio.run()` 里创建，pool 绑定的是那个已关闭的 event loop，在 PTB 的新 event loop 里使用时，asyncpg 内部检测到 event loop 不匹配，抛出：

```
asyncpg.exceptions.InterfaceError: cannot perform operation: another operation is in progress
```

**错误模式（之前的写法）：**
```python
# asyncio.run() 创建 event loop A，_setup() 在 A 里建 pool，A 随后被关闭
app, pool = asyncio.run(_setup())

# run_polling() 在内部创建 event loop B（全新的）
# 此后所有 handler 都在 B 里运行
# 但 pool 绑定的是已死的 A → 随机出现连接错误
app.run_polling(...)
```

**正确解法：使用 `post_init` 在 PTB 的 event loop 内创建 pool：**

```python
async def _post_init(application: Application) -> None:
    # 此回调由 PTB 在自己的 event loop 内调用
    # pool 绑定到 PTB 的 event loop，生命周期一致
    pool = await asyncpg.create_pool(_DB_DSN, min_size=5, max_size=20)
    application.bot_data["db_pool"] = pool

async def _post_shutdown(application: Application) -> None:
    pool = application.bot_data.get("db_pool")
    if pool:
        await pool.close()

# 启动时：只用临时单连接读配置，用完立即关闭
bot_token, chat_id, llm_config = asyncio.run(_get_startup_config())

app = (
    Application.builder()
    .token(bot_token)
    .post_init(_post_init)       # pool 在 PTB event loop 内创建
    .post_shutdown(_post_shutdown)
    .build()
)
app.run_polling(...)  # PTB 管理自己的 event loop，pool 与之绑定
```

**本质原因：asyncio 对象（连接、pool、socket）必须在创建它们的 event loop 内使用。** 跨 event loop 使用是 asyncio 的未定义行为。

### 6.4 并发控制：block=False + 任务队列

```python
# block=False：PTB 不等待 handler 完成就继续处理下一条消息
# 如果 block=True（默认），LLM 调用（30s+）期间 Bot 冻结，无法处理任何消息
app.add_handler(MessageHandler(
    filters.TEXT & ~filters.COMMAND,
    handle_message,
    block=False  # 关键：非阻塞，handler 作为 asyncio task 调度
))
```

但 `block=False` 会导致同一用户的多条消息并发处理，并发抢同一个 DB 连接。解决方案：每用户一个 `asyncio.Queue`，串行消费：

```python
# 每用户独立 queue + worker，保证串行
if chat_id not in queues:
    queues[chat_id] = asyncio.Queue(maxsize=10)
if chat_id not in workers or workers[chat_id].done():
    workers[chat_id] = asyncio.create_task(
        _queue_worker(chat_id, queue, context)
    )
await queue.put(update)  # 入队，worker 串行消费
```

### 6.5 上下文记忆

Bot 在内存里按 chat_id 维护对话历史，每条消息携带最近 10 轮（20 条 message）传给 LLM：

```python
history = histories.setdefault(chat_id, [])
# 成功后追加，超过 20 条滚动丢弃最早的（滑动窗口）
history.append({"role": "user", "content": user_text})
history.append({"role": "ai", "content": response_text})
if len(history) > 20:
    histories[chat_id] = history[-20:]
```

**为什么存内存而不存 DB？** 对话历史是临时状态，Bot 重启后自然清空，用户会重新开始对话，不需要持久化。存 DB 反而增加写入开销和读取延迟。

## 七、通知推送系统（深度解析）

### 7.1 站内通知 SSE 完整链路

```
Python stream_consumer
  评估完成 → INSERT notifications 表
           → Redis PUBLISH "notifications" channel（JSON payload）
                    │
                    ▼
Go notification_handler.StreamNotifications（SSE 端点）
  订阅 Redis "notifications" channel
  每条 Pub/Sub 消息 → 格式化为 SSE 帧 → Flush 到浏览器
  每 30 秒发送心跳帧（防止代理断开空闲连接）
  响应头：X-Accel-Buffering: no（防止 nginx 缓存 SSE 帧）
                    │
                    ▼
前端 NotificationBell.vue
  EventSource 长连接（GET /api/notifications/stream）
  收到 type=notification → 铃铛角标 +1 → 下拉列表追加
  收到 type=heartbeat → 无操作（保活确认）
```

**为什么用 Redis Pub/Sub 而不是直接写 Go 内存队列？**

Python 和 Go 是两个独立进程。Python 完成评估后没有办法直接调用 Go 内存里的数据结构。Redis Pub/Sub 是进程间通信的最简方案：Python PUBLISH，Go 已订阅的 goroutine 立即收到，不需要 HTTP polling。

### 7.2 Telegram 主动推送链路

```
stream_consumer._create_notification()
  → INSERT notifications（站内通知）
  → Redis PUBLISH（SSE 推送给浏览器）
  → push_service.push_to_channels()
      → 从 notification_settings.push_channels JSONB 读取所有渠道
      → 对每个 enabled=true 的 telegram 渠道：
          _push_telegram() → Telegram Bot API /sendMessage
          格式：MarkdownV2（18 种特殊字符必须 \\ 转义，否则 API 拒绝）
```

**推送触发条件（通过 DB 配置，热更新无需重启）：**
1. `notify_on_interesting=true` 且 `decision=INTERESTING`
2. OR `innovation_score >= min_innovation_score AND depth_score >= min_depth_score`

两条规则取 OR，用户可以用"高分阈值"兜底捕获非 INTERESTING 但仍高分的文章。

## 八、数据库设计

### 核心表关系

```
sources (1) ──── (N) content (1) ──── (1) evaluation
                      │
                      └── status 状态机：
                          PENDING → PROCESSING → EVALUATED
                                              ↘ DISCARDED（eval_attempts >= 3）
```

### 核心表字段设计

**content 表（关键字段）：**
```sql
id             BIGSERIAL PRIMARY KEY
source_id      BIGINT REFERENCES sources(id) ON DELETE CASCADE
original_url   VARCHAR(500) UNIQUE NOT NULL  -- URL 唯一约束，三级去重的 L3
title          TEXT
content        TEXT
status         VARCHAR(20) DEFAULT 'PENDING'
eval_attempts  INTEGER DEFAULT 0             -- 防死循环
created_at     TIMESTAMP DEFAULT NOW()
```

**evaluation 表：**
```sql
id               BIGSERIAL PRIMARY KEY
content_id       BIGINT REFERENCES content(id) UNIQUE  -- 一对一
innovation_score INTEGER                                -- 0-10
depth_score      INTEGER                                -- 0-10
decision         VARCHAR(20)                            -- INTERESTING/BOOKMARK/SKIP
tldr             TEXT
reasoning        TEXT
key_concepts     JSONB                                  -- 数组存为 JSONB
```

**notification_settings 表（推送配置）：**
```sql
id                  INTEGER PRIMARY KEY DEFAULT 1       -- 单行设计
enabled             BOOLEAN DEFAULT true
push_channels       JSONB                               -- 渠道数组，扩展无需改表
min_innovation_score INTEGER DEFAULT 8
min_depth_score      INTEGER DEFAULT 7
notify_on_interesting BOOLEAN DEFAULT true
```

### 关键设计决策

**`push_channels` 用 JSONB 数组而非单独表：**
推送渠道配置变化频率低，JSONB 避免多表 JOIN，前端直接传 JSON 对象数组，新增渠道类型（比如 Email、Slack）无需迁移表结构，只需前后端约定新的 `type` 字段值。

**`ai_config` 单行表：**
全局只有一套 LLM 配置，用 `id=1` 单行存储，`ON CONFLICT DO UPDATE` 做 upsert，简单高效，避免多行配置的版本管理问题。

**`eval_attempts` 防死循环：**
LLM 评估失败时递增，达到阈值（3次）自动 DISCARDED，防止格式异常的文章无限占用队列资源。

### 8.2 Pinia Store 设计

**useConfigStore — 配置的双写策略**

LLM 配置同时存在两个地方：localStorage（本地备份）和 PostgreSQL（供 stream_consumer 使用）。启动时两者同步：

```javascript
const loadConfig = async () => {

    const saved = localStorage.getItem('config')
    if (saved) { /* 恢复 apiKey, modelName, baseUrl */ }

    const data = await fetch(`/api/config/llm`).then(r => r.json())
```

### 8.3 SSE 双路实现

前端存在两套 SSE 实现，用途不同：

**路径 A：useSSE.js（EventSource，GET SSE）**

用于订阅 Go 后端的实时通知推流（`GET /api/notifications/stream`）：

```javascript
// EventSource 原生支持 GET，自动断线重连
eventSource = new EventSource('/api/notifications/stream')
eventSource.addEventListener('message', (event) => {
    const data = JSON.parse(event.data)
    // 新文章评估完成 → 触发 Toast + 铃铛动效
})
```

**路径 B：useAPI.js（fetch + ReadableStream，POST SSE）**

用于 Agent 对话（`POST /api/agent/chat`），需要在 body 里传 LLM 配置，EventSource 只支持 GET，所以用 fetch 手动消费流：

```javascript
const response = await fetch('/api/agent/chat', {
    method: 'POST',
    body: JSON.stringify({ message, llm_config }),  // 必须 POST 携带配置
})

const reader = response.body.getReader()
const decoder = new TextDecoder()
let buffer = ''

while (true) {
    const { done, value } = await reader.read()
    if (done) break
    buffer += decoder.decode(value, { stream: true })

    // 按行解析（chunk 可能在任意位置被截断）
    const lines = buffer.split('\n')
    buffer = lines.pop()  // 最后一行可能不完整，留到下一个 chunk
    for (const line of lines) {
        if (!line.startsWith('data: ')) continue
        const data = JSON.parse(line.slice(6))
        if (data.type === 'chunk')     appendChar(data.content)     // 打字机追加
        if (data.type === 'tool_call') showAgentStep(data)          // 工具步骤展开
        if (data.type === 'done')      finishStreaming()
        if (data.type === 'error')     showError(data.error)
    }
}
```

**为什么要 buffer 处理？**
TCP 不保证数据按 SSE 事件边界到达，一个 `reader.read()` 可能只读到半行。buffer 保留上次未完整的行，与下次读取的数据拼接后再解析。

## 九、运维与可靠性

### 进程独立性

三个 Python 进程完全独立，任何一个崩溃不影响其他：

| 进程 | 崩溃影响 | 恢复方式 |
|------|---------|------|
| api_server | 前端 Agent 对话不可用，其他正常 | 重启即可，无状态 |
| stream_consumer | 新文章停止评估，已有数据不受影响 | 重启后 pending 消息自动重投 |
| telegram_bot | Bot 不响应，评估推送暂停 | 重启即可，历史记录在内存丢失（可接受） |

### Redis Stream 可靠性保证

| 特性 | 实现 |
|------|------|
| 至少一次处理 | XREADGROUP 后才 ACK，崩溃时消息留 pending list |
| 重复处理幂等 | DB UNIQUE 约束，重复 INSERT 被忽略 |
| Pending 重投 | 每次循环开头调 `_requeue_pending_content` 检查 |
| 消息积压可见 | `XLEN` 查 Stream 长度，`XPENDING` 查待处理消息 |

### 容错链路完整性

```
LLM 评估失败
  → eval_attempts 递增
  → retry（最多 3 次，LangGraph 条件边）
  → 超限标记 DISCARDED，不再重试

Redis ACK 延迟
  → 消息在 pending list
  → Consumer 重启后 XREADGROUP 重新投递
  → DB UNIQUE 约束保证幂等

DB 写入失败
  → 异常被 catch，记录日志
  → 消息不 ACK，留 pending 等待重试
```

## 十一、真实难点与解决过程（面试亮点）

### 难点 1：asyncpg Event Loop 错配导致命令失败

**现象：** Telegram Bot 的 `/fetch`、`/recent` 命令持续报 `cannot perform operation: another operation is in progress`，而 `/status`（只用 aiohttp，不用 DB）正常。

**排查过程：**
1. 初步怀疑连接池耗尽 → 加大 max_size 到 20 → 无效
2. 怀疑并发写冲突 → 加任务队列串行化 → 无效
3. 怀疑工具代码连接泄漏 → 审查 tools.py → 无异常
4. 最终定位：`asyncio.run(_setup())` 创建 event loop A 并关闭，`app.run_polling()` 创建 event loop B，asyncpg pool 绑定已死的 A

**解决：** 使用 `post_init` 在 PTB 自己的 event loop 内创建 pool，启动配置用临时单连接读取后立即关闭。

**核心教训：** asyncio 对象必须在创建它们的 event loop 内使用。跨 event loop 共享是未定义行为，symptoms 可能随机、难以复现。

### 难点 2：Bot 处理第一条消息后冻结

**现象：** Bot 成功处理第一条自然语言消息后，后续所有消息无响应。

**原因：** `MessageHandler` 默认 `block=True`，LLM 调用（30s+）期间 handler 阻塞，PTB 的 update 处理循环卡死，后续所有消息入队但无法调度。

**解决：** `MessageHandler(..., block=False)` + `asyncio.wait_for(..., timeout=60.0)` 防止单条消息无限阻塞。

### 难点 3：第三方 LLM Relay 兼容性问题

**现象：** 部分中转站非流式模式下，LLM 返回只有 `reasoning` 字段，`content` 为 null，导致 parse_node 解析失败、eval_attempts 耗尽、文章被 DISCARDED。

**解决：** `content_evaluator.py` 强制 `streaming=True`，逐 chunk 拼接完整输出，规避 relay 的非流式 bug。

### 难点 4：arXiv RSS 返回 892B（异常小）

**现象：** arXiv 的 RSS 端点返回 200 但只有 892B，正常应有数百 KB。

**分析：** arXiv 对 curl UA 做了限流或内容限制。不添加该源，避免抓取到空内容浪费 LLM Token。

## 十二、设计决策速查（面试准备）

| 问题 | 答案要点 |
|------|------|
| 为什么用 Redis Stream 而不是 Kafka？ | 个人项目，Redis 已有，Stream 功能足够；Kafka 运维成本高，过度工程 |
| Bloom Filter 误判怎么办？ | 误判只会把新内容当重复（漏抓），极低概率（<0.1%）且可接受；不会漏判（不会把重复当新的） |
| 为什么三级去重而不是只用 DB UNIQUE？ | 性能；单用 DB 每条都打 IO；三级让 99.9% 在内存拦截 |
| LangGraph 能换成简单 while 循环吗？ | 能；LangGraph 的优势是状态可视化和测试，逻辑等价于带状态的 while loop |
| SSE 和 WebSocket 如何选择？ | 单向推送用 SSE（简单、HTTP/1.1 兼容、自动重连）；双向实时用 WebSocket |
| 为什么不把 LLM 配置放 .env？ | 配置热更新需求：stream_consumer 每 60 秒从 DB 加载，无需重启进程 |
| 推送通知为什么用 JSONB 而非单独表？ | 渠道类型扩展不需要改表结构；配置变更频率低，不需要关系型查询 |

## 十三、数据库 Schema 设计决策

### 13.1 content / evaluation 分表设计

```sql
-- content 表：文章生命周期状态机
CREATE TABLE content (
    id           SERIAL PRIMARY KEY,
    source_id    INTEGER REFERENCES sources(id),
    title        TEXT NOT NULL,
    original_url TEXT UNIQUE NOT NULL,  -- 全局唯一约束：三级去重的最终防线
    content      TEXT,
    image_urls   JSONB DEFAULT '[]',
    status       TEXT DEFAULT 'PENDING', -- PENDING / PROCESSING / EVALUATED / DISCARDED
    created_at   TIMESTAMP DEFAULT NOW()
);

-- evaluation 表：LLM 评估结果（1:1 with content）
CREATE TABLE evaluation (
    id               SERIAL PRIMARY KEY,
    content_id       INTEGER UNIQUE REFERENCES content(id),
    innovation_score INTEGER,
    depth_score      INTEGER,
    decision         TEXT,   -- INTERESTING / NOT_INTERESTING
    tldr             TEXT,
    reasoning        TEXT,
    eval_attempts    INTEGER DEFAULT 0,
    created_at       TIMESTAMP DEFAULT NOW()
);
```

**为什么分表而不合并？**

| 维度 | 合并的代价 | 分表的收益 |
|------|-----------|-----------|
| 查询效率 | content 列表查询带上大段 reasoning TEXT，无谓传输 | SELECT c.* 与 JOIN e.* 各司其职，按需 JOIN |
| 职责分离 | 评估字段可能 NULL，语义混乱 | content 表永远完整；evaluation 有则精确 |
| 索引策略 | 所有列在一张大表，索引选择性低 | decision/score 索引只在 evaluation 表，小而快 |
| 未来扩展 | 多版本评估（A/B 测试）需大量 NULL 列 | evaluation 表可多行记录同一 content 的多次评估 |

### 13.2 eval_attempts = 3 的设计理由

```python
# stream_consumer.py 中的评估状态机
MAX_EVAL_ATTEMPTS = 3

while article.eval_attempts < MAX_EVAL_ATTEMPTS:
    result = await evaluator.evaluate(article)
    if result.is_valid():
        break
    article.eval_attempts += 1

if article.eval_attempts >= MAX_EVAL_ATTEMPTS:
    article.status = "DISCARDED"  # 超过重试次数直接丢弃
```

**为什么是 3 次？**
- LLM 单次失败率（格式错误/API 超时）约 5-10%
- 3 次重试后失败概率 = 0.1³ = 0.001（0.1%），可接受
- 超过 3 次继续重试：成本线性增长，但对低质量内容意义不大
- 1 次：5-10% 丢弃率太高；5 次：99.999% 成功但成本加倍

**实际触发场景：**
1. LLM 返回格式错误（非 JSON/缺字段） → parse_node 失败 → 重试
2. API 超时（第三方 relay 不稳定） → 捕获异常 → 重试
3. reasoning 为 null（部分 relay 的 bug） → 强制 streaming=True 后已基本消除

### 13.3 push_channels JSONB 数组设计

```sql
CREATE TABLE notification_settings (
    id            SERIAL PRIMARY KEY,
    push_channels JSONB DEFAULT '[]',  -- 多渠道配置数组
    ...
);

-- 实际存储结构
[
  {
    "type": "telegram",
    "enabled": true,
    "bot_token": "7xxx:AAA...",
    "chat_id": "813xxx",
    "notify_on_interesting": true,
    "min_score_threshold": 7
  }
  -- 未来可扩展：{"type": "email"}, {"type": "webhook"}, ...
]
```

**JSONB vs 关系型 `notification_channels` 表的权衡：**

| 考量 | JSONB 方案 | 关系型表方案 |
|------|-----------|------------|
| 渠道字段差异 | Telegram 有 bot_token，Email 有 smtp_host，字段完全不同，JSONB 自然适配 | 需要 EAV 模式或宽表，大量 NULL |
| 查询模式 | 每次推送读取全部配置（单行全量读取），无需 WHERE 过滤 | JOIN + WHERE 反而增加复杂度 |
| 变更频率 | 配置极少变更（用户偶尔添加/删除渠道），不需要高并发写 | 关系型优势在高频写入，此场景不适用 |
| 扩展成本 | 新增渠道类型：零改表，新增文档即可 | 需要 ALTER TABLE 或新建 channel_type 子表 |

**缺点：** JSONB 内的字段无类型约束，需在应用层校验。本项目通过前端 Config.vue 的表单校验保证。

### 13.4 ai_config 单行设计

```sql
CREATE TABLE ai_config (
    id            SERIAL PRIMARY KEY,  -- 永远只有 id=1 这一行
    default_model TEXT DEFAULT 'glm-4-flash',
    api_key       TEXT DEFAULT 'sk-placeholder',
    base_url      TEXT DEFAULT 'https://open.bigmodel.cn/api/paas/v4/',
    temperature   NUMERIC(3,2) DEFAULT 0.7,
    max_tokens    INTEGER DEFAULT 2000,
    updated_at    TIMESTAMP DEFAULT NOW()
);
```

**为什么用单行而不是 key-value 表？**

个人工具，永远只有一套 LLM 配置。如果改成 key-value：

```sql
-- 反例：过度设计
CREATE TABLE config_kv (key TEXT PRIMARY KEY, value TEXT);
-- 读取时需要 SELECT * FROM config_kv WHERE key IN ('api_key', 'model', ...)
-- 再手动 MAP 到结构体 → 代码冗余，类型信息丢失
```

单行方案：`SELECT * FROM ai_config LIMIT 1` → 直接映射到结构体，类型安全。

**热加载实现：** stream_consumer 后台线程每 60 秒执行此查询，无需重启进程即可生效新的 API key 或模型切换。

## 十四、性能量化 & 成本估算

### 14.1 三级去重的成本节省推导

**假设基线：** 73 个 RSS 源，每 30 分钟全量抓取，每源平均 30 条，单次抓取 2190 条原始数据。

**历史数据分布（实测估算）：**
- 相同 URL 重复抓取率：约 85%（同一篇文章在 30 分钟窗口内多次出现）
- 跨源重复率：约 3%（技术媒体互相转载）
- 真正新内容：约 12%，即约 263 条/轮

**三级过滤效果：**

| 层级 | 拦截位置 | 拦截率 | 后续处理节省 |
|------|---------|--------|------------|
| L1：Bloom Filter | 内存（Go 进程内） | ~80%（已见过的 URL） | 省去 Redis 网络 RTT × 1750 次 |
| L2：Redis Set | 网络（单次 SISMEMBER） | ~15%（L1 漏过的旧 URL） | 省去 PostgreSQL INSERT 尝试 |
| L3：DB UNIQUE | 磁盘（PostgreSQL） | <1%（Race condition 兜底） | 防止数据重复 |

**LLM 调用节省：**
- 无去重：2190 次 LLM 调用/轮
- 有去重：~263 次 LLM 调用/轮（节省 88%）
- 按 1000 tokens/次、GLM-4-Flash ¥0.1/M tokens 计算：每轮节省约 ¥0.19，每天（48轮）节省约 ¥9.1
- "节省 30%"的保守说法基于：即使只有 70% 是重复，3 层过滤也能拦住这些调用

### 14.2 Bloom Filter 内存计算

目标：1M URL，误判率（FPR）≤ 0.1%

**公式：**
```
最优 bit 数 m = -n × ln(p) / (ln2)²
最优哈希函数数 k = (m/n) × ln2

n = 1,000,000（预期 URL 数）
p = 0.001（目标误判率）

m = 1,000,000 × 6.908 / 0.480 ≈ 14,400,000 bits ≈ 1.72 MB
k = (14,400,000 / 1,000,000) × 0.693 ≈ 10 个哈希函数
```

**实际开销：~1.72 MB 内存，换取 0 次网络 IO（对已见 URL）**

对比：Redis Set 中每个 URL 约 50-100 bytes，1M URL 约 50-100 MB RAM + 网络 RTT。Bloom Filter 以 1/30 的内存代价实现等效功能（允许极低误判率）。

### 14.3 实际吞吐瓶颈分析

**当前系统（73源，本地运行）：**

```
抓取层（Go）：73 源 × ~5s/源（含网络） / 50并发 ≈ 8秒/轮
               实际瓶颈：网络 I/O，协程池上限 50

过滤层：Bloom Filter < 1ms/条，Redis SISMEMBER ~0.5ms/条
        对 2190 条：总计 < 2 秒，不是瓶颈

评估层（Python）：每篇文章 3-10 秒（含 LLM API 调用）
                 263 新文章 × 6s平均 = 26 分钟
                 ⚠️ 这里是实际瓶颈
```

**结论：** 在 30 分钟抓取周期内，LLM 评估层理论上来得及处理所有新文章（~26 分钟 < 30 分钟）。但抓取量突增（如 RSS 源爆发性更新）时，积压会在 Redis Stream 中缓冲。

**如果扩展到 500 个源：**
- 新内容约增加至 ~1800 条/轮
- LLM 评估时间：1800 × 6s = 180 分钟（远超 30 分钟抓取周期）
- Stream 积压持续增长，需要水平扩展消费者

## 十五、偏好注入全链路

### 15.1 完整数据流（含时延）

```
用户输入（Telegram Bot 或前端 Agent）
    "我不喜欢 Rust 相关的文章"
         │  <1 秒：LLM 解析意图
         ▼
ReAct Agent → update_preferences 工具调用
         │  <50ms：asyncpg 写入 PostgreSQL
         ▼
user_preferences 表（source_id IS NULL 为全局偏好）
{"disliked_topics": ["Rust"]}
         │
         │  ← 时延窗口：0 到 60 秒（下次评估触发前）
         ▼
stream_consumer._inject_preferences()
    - 读取全局偏好（SELECT preferences WHERE source_id IS NULL）
    - 若当前文章有源级偏好（source_id 匹配），优先使用源级
         │
         ▼
format_preferences_for_prompt() 格式化：
    "用户偏好：
     不感兴趣的话题：Rust
     请在评估时考虑以上偏好，适当调整评分。"
         │
         ▼
动态拼接到 system_prompt 末尾 → LLM 评分
    Rust 相关文章 innovation/depth 分数降低 1-3 分
    decision 可能从 INTERESTING 变为 NOT_INTERESTING
```

### 15.2 源级偏好 vs 全局偏好（优先级规则）

```python
async def _inject_preferences(content_id: int, source_id: int, db_pool) -> str:
    # 先查源级偏好
    row = await db_pool.fetchrow(
        "SELECT preferences FROM user_preferences WHERE source_id = $1", source_id
    )
    if row and row["preferences"]:
        return format_preferences_for_prompt(row["preferences"])
    
    # 回退全局偏好
    row = await db_pool.fetchrow(
        "SELECT preferences FROM user_preferences WHERE source_id IS NULL"
    )
    if row and row["preferences"]:
        return format_preferences_for_prompt(row["preferences"])
    
    return ""  # 无偏好配置，不注入
```

**设计意图：** 某些 RSS 源有独特的内容风格（如安全漏洞源，深度要求更高），可单独配置偏好覆盖全局默认值。

**冲突解决：** 不做合并，源级完全覆盖全局。避免"全局不喜欢 Rust，但该源专注 Rust 系统编程"的矛盾注入。

### 15.3 增量合并策略（数组字段）

```python
# 用户先说："我喜欢 AI 文章"
# 后说："我喜欢系统设计"
# 结果：liked_topics = ["AI", "系统设计"]，不覆盖

array_fields = ["liked_topics", "disliked_topics"]
for key, value in new_prefs.items():
    if key in array_fields:
        existing_list = existing.get(key, [])
        for item in value:
            if item not in existing_list:
                existing_list.append(item)  # 追加去重
        existing[key] = existing_list
    else:
        existing[key] = value  # 标量字段（min_score）直接覆盖
```

**为什么数组追加而标量覆盖？**
- `liked_topics` 是偏好集合，用户每次表达的是增量偏好
- `min_innovation_score` 是阈值，用户后来的设置代表当前意图（"改成 7 分"表示替换）

### 15.4 实时生效 vs 延迟生效

| 操作 | 生效时机 | 原因 |
|------|---------|------|
| 偏好写入 DB | 立即 | asyncpg 写操作，<50ms |
| 当前正在评估的文章 | 不受影响 | 已经读取了旧 system_prompt |
| 下一篇文章评估 | 立即生效 | _inject_preferences 每次评估前调用 |
| stream_consumer LLM 配置重载 | 最多 60 秒 | 后台线程定时重载 LLM 配置（≠偏好配置） |

**偏好配置每次评估前实时读取，无缓存，无需重启任何进程。**

## 十六、前端架构选型

### 16.1 Vue 3 vs React 的实际决策

项目启动时（2025 Q4），选 Vue 3 而非 React 的考量：

| 维度 | Vue 3 | React | 本项目的权衡 |
|------|-------|-------|------------|
| 学习曲线 | 渐进式，SFC 上手快 | Hooks 心智负担高（闭包陷阱、依赖数组） | 个人项目，减少调试时间 |
| 响应式系统 | `ref`/`reactive` 直观，无需手动声明依赖 | useState 需要手动管理，容易过时闭包 | 偏好系统等状态变更频繁，Vue 响应式更简洁 |
| 模板语法 | `v-if`/`v-for` 声明式，HTML 结构清晰 | JSX 强大但混合逻辑与视图 | RSS 阅读器有复杂条件渲染，模板更直观 |
| Pinia vs Redux | Pinia API 极简（defineStore + state + actions） | Redux Toolkit 仍有 boilerplate | 多个 store（config/timeline/notification）用 Pinia 更轻 |
| 生态 | Vite 原生支持，Tailwind 配置简单 | 同样支持，无差异 | 无差异 |

**结论：** 对个人工具类项目，Vue 3 在开发效率上有优势。React 在团队规模大、组件复用需求高时更合适。

### 16.2 SSE 双路径设计

前端使用了两种截然不同的 SSE 实现，分别对应不同场景：

**路径 A：EventSource（通知铃铛）**
```javascript
// useSSE.js
const es = new EventSource('http://localhost:8080/api/notifications/stream')
es.onmessage = (event) => {
  const notification = JSON.parse(event.data)
  notificationStore.add(notification)
}
```
- 适合：长连接、自动重连、不需要请求体
- 限制：只能 GET，无法携带 POST body

**路径 B：fetch + ReadableStream（Agent 对话）**
```javascript
// useAPI.js
const response = await fetch('/api/agent/chat', {
  method: 'POST',
  body: JSON.stringify({ message, llm_config }),
  headers: { 'Content-Type': 'application/json' }
})
const reader = response.body.getReader()
while (true) {
  const { done, value } = await reader.read()
  if (done) break
  // 解析 SSE 格式的 chunk
  parseSSEChunk(decoder.decode(value))
}
```
- 适合：需要 POST body（LLM 配置、消息历史）
- 代价：需要手动处理断线重连

**选择依据：** 通知推送是系统主动发，不需要请求体 → EventSource 最简单。Agent 对话需要携带 LLM 配置和历史上下文 → 必须用 fetch POST。

### 16.3 localStorage + DB 双层配置同步

Config 页面的状态同步策略：

```javascript
// useConfigStore.js
const configStore = defineStore('config', {
  state: () => ({
    llmConfig: null,    // 从 DB 加载
    sources: [],        // 从 DB 加载
  }),
  actions: {
    async loadAll() {
      // 优先从 DB 读取（权威数据源）
      const [llm, sources] = await Promise.all([
        api.getLLMConfig(),
        api.getSources()
      ])
      this.llmConfig = llm
      this.sources = sources
    },
    async saveLLMConfig(config) {
      await api.postLLMConfig(config)  // 写 DB（持久化）
      this.llmConfig = config          // 更新本地状态
    }
  }
})
```

**不使用 localStorage 持久化配置的原因：**
- LLM API key 等敏感信息不应存 localStorage（XSS 风险）
- 多设备访问时 localStorage 会不同步
- DB 是权威数据源，避免 localStorage 与 DB 不一致时的冲突解决逻辑

**PRESET_SOURCES 的惰性检查：**
```javascript
// 判断推荐源是否已添加（避免重复添加）
const isPresetAdded = (url) => {
  return configStore.sources.some(s => s.url === url)
}
```
响应式依赖 `configStore.sources`，删除后自动变回"未添加"状态。

```
App.vue
├── NotificationBell.vue    ← useNotificationStore（全局 SSE 监听）
```

## 十七、系统扩展性分析

### 17.1 当前单机瓶颈（500+ 源场景）

**抓取层（Go）：**
- 现有：协程池上限 50，73 源下约 8 秒/轮
- 500 源：约 55 秒/轮（仍在 30 分钟周期内）
- 1000 源：约 110 秒/轮，接近极限但理论可行
- 真正瓶颈：Go 进程内存（每协程约 4-8KB，50 协程 < 1MB，不是问题）

**评估层（Python，真正瓶颈）：**
- 现在：73 源 → ~263 新文章/轮 → ~26 分钟（LLM 串行）
- 500 源（线性假设）：~1800 新文章/轮 → ~180 分钟，积压持续增长
- Stream 充当缓冲，短期峰值可以吸收，但长期积压无解

### 17.2 Redis Stream Consumer Group 水平扩展

```
                    Redis Stream: ingestion_queue
                           │
              ┌────────────┼────────────┐
              ▼            ▼            ▼
         Consumer-1    Consumer-2    Consumer-3
         (Python进程)  (Python进程)  (Python进程)
              │            │            │
              └────────────┴────────────┘
                           │
                      PostgreSQL
```

**只需改一行代码：**
```python
# 现在（单消费者）
await redis.xreadgroup("junkfilter-group", "consumer-1", {"ingestion_queue": ">"}, count=1)

# 扩展后（多消费者，consumer-N 不同即可并行）
await redis.xreadgroup("junkfilter-group", f"consumer-{os.getpid()}", {"ingestion_queue": ">"}, count=1)
```

Redis Consumer Group 保证：同一消息只被一个 Consumer 消费，自动负载均衡，无需协调层。

**其他需要处理的问题：**
- DB 写入并发：asyncpg pool 已经支持，PostgreSQL 可处理高并发写入
- LLM API 并发限制：第三方 relay 通常有 RPM 限制，消费者数量需与 API 限额匹配

### 17.3 多用户支持最小改造

当前硬编码单用户的位置：

| 位置 | 当前 | 改造成本 |
|------|------|---------|
| `user_preferences` 表 | `source_id IS NULL` 为全局唯一偏好 | 增加 `user_id` 字段，查询加 WHERE user_id = ? | 低 |
| Telegram Bot 白名单 | `allowed_ids = {"813...", "658..."}` 硬编码 | 改为 DB 表 `authorized_users(chat_id, user_id)` | 低 |
| `notification_settings` 表 | `id=1` 单行 | 增加 `user_id` 字段 | 低 |
| `ai_config` 表 | 全局共用一套 LLM 配置 | 每用户可选独立配置（需 API key 分离） | 中 |
| RSS `sources` 表 | 所有用户共享同一套 RSS 源 | 增加 `user_id` 字段，或保留共享模式 | 中 |
| 认证层 | 无 | JWT + 用户注册登录（Go 中间件） | 高 |

**最小可行多用户（MVP）：**
1. 增加 `users` 表 + JWT 认证中间件（Go）
2. `user_preferences` + `notification_settings` 加 `user_id` 列
3. sources 保持共享（公共订阅池）
4. Telegram Bot 白名单改为 DB 查询

**不需要改的部分：** Redis Stream 结构、LangGraph 评估逻辑、Go 抓取层（源共享）、SSE 推送机制。
